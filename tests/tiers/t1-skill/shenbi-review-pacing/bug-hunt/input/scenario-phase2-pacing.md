# 行为测试：节奏审计 — 连续蓄压超限

测试目标：验证 `shenbi-review-pacing` 技能能在章节中检测出**连续 QUEST 章超过 `maxConsecutiveQuest` 上限**的停滞风险。

## 测试场景

本测试提供一组 4 个连续 QUEST 章节的场景（章节 N-3、N-2、N-1、N），其中**刻意植入**了一个节奏 bug：4 个连续 QUEST 章节超过了 `genre-config.json` 中设定的 `maxConsecutiveQuest: 3` 上限，期间没有任何 FIRE 章节释放。正确加载节奏审计技能的 agent 必须**报告这个超限**为 `warning` 级别（按铁律1：停滞警告），并给出具体修复建议。

## 准备

- 小说项目存在 `truth/chapter_summaries.md` 和 `genre-config.json`
- `genre-config.json` 中 `pacingRules`：
  - `maxConsecutiveQuest: 3`（本测试项目定制值，spec 默认 5）
  - `maxGapQuest: 3`
- `truth/chapter_summaries.md` 记录前 3 章类型：
  - 第 N-3 章：QUEST — 主角独自追查"失踪弟子的线索"
  - 第 N-2 章：QUEST — 主角进入疑似的废弃矿脉搜索
  - 第 N-1 章：QUEST — 主角在矿脉深处发现魔修仪式痕迹

## 输入（章节 N 节选）

```markdown
# 第 N 章 顺藤摸瓜

矿脉深处寒气森然。

林轩沿着残余的符文刻痕，蹲在地上仔细辨认了半个时辰，终于拼出一条清晰的撤退路线——魔修是往北云山方向撤的。

他站起身，拂去膝上的灰尘。三天前他追踪"失踪弟子"的线索至此，两天前他进入矿脉搜索，昨天他在深处发现了魔修仪式的痕迹。每一步都在向前推进，每一步都离真相更近一点——但也每一步都离危险更近一点。

"再追下去，还是先回报门派？"他低声自问。

沉默良久，他做下了决定：继续追。他知道这不符合师父"谋定后动"的叮嘱，也知道掌门"封锁区域"的禁令。但线索正在冷却，魔修正在远去。

他按了按腰间的长刀，朝北云山方向迈出了步子。
```

## 植入的 Bug

| 章节 | 类型 | 摘要 | 蓄压/爆发 |
|------|------|------|----------|
| N-3 | QUEST | 主角独自追查"失踪弟子的线索" | 蓄压 |
| N-2 | QUEST | 主角进入疑似的废弃矿脉搜索 | 蓄压 |
| N-1 | QUEST | 主角在矿脉深处发现魔修仪式痕迹 | 蓄压 |
| N   | QUEST | 主角决定顺藤摸瓜，继续独自追踪 | 蓄压（本章） |

**问题**：连续 4 章 QUEST 无任何 FIRE 释放，超过 `maxConsecutiveQuest: 3` 上限。按铁律1（蓄压必须有释放），触发"停滞警告"。

## 期望的审计输出

```markdown
## 节奏审计报告

**章节**: 第N章
**本章类型**: QUEST
**结果**: 有瑕疵

### 近5章类型序列
| 章节 | 类型 | 蓄压/爆发状态 |
|------|------|-------------|
| N-3 | QUEST | 蓄压 |
| N-2 | QUEST | 蓄压 |
| N-1 | QUEST | 蓄压 |
| N   | QUEST | 蓄压（本章） |

### 规则检查
- maxConsecutiveQuest: 4/3 超出上限 1
- maxGapFIRE: 4/3 超出上限 1
- 序列多样性: QUEST 100% WARNING（近4章无任何类型变化）

### 评分: 5/10 有瑕疵

### 建议修复
- [WARNING] 连续 4 章 QUEST 无 FIRE（超过 maxConsecutiveQuest: 3 上限）：本章或下一章必须安排一个 FIRE 章节作为释放——例如主角在追踪途中遭遇魔修伏击并展开战斗，或发现关键证据导致局势逆转
```

## 通过条件

- [ ] 审计报告识别出"连续 4 章 QUEST 无 FIRE"违反 `maxConsecutiveQuest: 3` 规则
- [ ] 标记为 `WARNING` 级别（按铁律1：蓄压必须有释放 → 停滞警告）
- [ ] 章节评分 ≤ 6/10（有瑕疵但非阻塞）
- [ ] 给出具体修复建议（指出应插入 FIRE 章节作为释放）

## 失败条件

- 未检测出连续 QUEST 超限 → FAIL
- 检测出但未标注 `WARNING` 级别（按铁律1应为 warning） → FAIL
- 报告含糊未指明具体规则违反（`maxConsecutiveQuest: 3`） → FAIL
- 未给出修复建议 → FAIL
- 用"读者有耐心"或"蓄压也行"等理由放过此超限 → FAIL
