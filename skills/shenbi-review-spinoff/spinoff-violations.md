# 衍生作品违规类别

此参考由 `shenbi-review-spinoff` 技能使用。详细说明 4 类违规（原作事件冲突 / 未来信息泄漏 / 世界规则不一致 / 伏笔隔离破坏）的判定方法。

## 1. 原作事件冲突

### 1.1 冲突类型

| 类型 | 描述 | 判定 |
|------|------|------|
| 状态矛盾 | 原作中角色存活/状态 vs 衍生中状态 | error |
| 时间矛盾 | 事件顺序在原作 vs 衍生不一致 | error |
| 关系矛盾 | 角色关系在原作 vs 衍生不一致 | error |
| 行为矛盾 | 原作明确行为 vs 衍生中同一角色另一行为 | error |
| 能力矛盾 | 原作能力 vs 衍生中能力 | error |

### 1.2 判定方法

1. 读取 `truth/parent_canon.md`（原作事件表 / 角色状态表）
2. 提取本章所有具体陈述（事件 / 角色状态 / 关系）
3. 逐项对比
4. 不在原作表中的"原创"内容 = pass（衍生本应扩展）
5. 与原作表矛盾 = error

### 1.3 边界情况

- 原作未明确的状态（如"角色 A 的童年"未写）= 衍生可补完 = pass
- 原作已明确，衍生中矛盾 = error
- 原作留白但有强烈暗示 = 衍生可补完，但需"暗示一致"

## 2. 未来信息泄漏

### 2.1 泄漏类型

| 类型 | 描述 | 严重度 |
|------|------|--------|
| 时点泄漏 | 衍生角色在原作"未揭示时点"知道"后揭示信息" | error |
| 全知泄漏 | 衍生作品叙述者展现原作读者才有的全知视角 | error |
| 跨时空对话 | 衍生角色与原作"未来角色"对话 / 引用未来事件 | error |

### 2.2 判定方法

1. 读取 `parent_canon.md` 的"信息揭示时点表"
2. 提取本章所有"角色知道 X"类陈述
3. 对每个 X 查时点：
   - X 在衍生时点之前已揭示 = pass
   - X 在衍生时点之后揭示 = error
4. 叙述者视角检查：
   - 第三人称限制视角 = pass（不越界）
   - 突然展现角色内心 = 需检查是否该角色在原作时点有此信息

### 2.3 边界情况

- 角色独立推导（基于衍生时点已知信息）= pass
- 角色回忆过去 = pass（回忆内容需与原作一致）
- 角色预知 / 梦境 = 需在 `world/rules.md` 显式声明该能力

## 3. 世界规则不一致

### 3.1 不一致类型

| 类型 | 描述 | 严重度 |
|------|------|--------|
| 物理规则 | 原作世界的物理 / 因果 / 时间规则 | error |
| 能力体系 | 原作世界的能力 / 等级 / 代价 | error |
| 社会制度 | 原作世界的社会 / 政治 / 经济制度 | error |
| 地理规则 | 原作世界的地理 / 距离 / 物流 | error |

### 3.2 判定方法

1. 读取 `world/rules.md`（应包含 shared + spinoff-specific 两段）
2. 提取本章所有规则使用
3. 对每条规则使用检查：
   - 符合 shared 规则 = pass
   - 符合 spinoff-specific 规则 = pass
   - 与任何规则矛盾 = error
4. 衍生引入新规则 = 需在 `world/rules.md` 标注 `spinoff-specific` + 在 `parent_canon.md` 备案

### 3.3 边界情况

- 衍生作品的"独立小世界"（如衍生故事发生在原作未覆盖的角落）= 可独立规则，但需声明
- 衍生作品的"特定组织 / 地区" = 共享规则 + 局部变体，变体需声明

## 4. 伏笔隔离

### 4.1 隔离类型

| 类型 | 描述 | 严重度 |
|------|------|--------|
| 范围混淆 | 衍生伏笔与原作伏笔混入同一池 | error |
| 跨作品激活 | 衍生作品激活原作伏笔 | error（除非显式联动声明） |
| 跨作品兑现 | 衍生作品兑现原作伏笔 | error（除非显式联动声明） |
| 状态不同步 | 原作伏笔被衍生作品"消耗"但原作未更新 | error |

### 4.2 判定方法

1. 读取 `truth/pending_hooks.md`，确认所有伏笔有 `scope` 字段（`shared` / `spinoff`）
2. 提取本章所有伏笔操作（open/advance/resolve/defer）
3. 对每个操作检查：
   - 操作对象 = spinoff 伏笔 = pass
   - 操作对象 = shared 伏笔，但有显式跨作品联动声明 = pass
   - 操作对象 = shared 伏笔，无声明 = error
4. 状态同步：衍生作品消耗 shared 伏笔后，原作 `pending_hooks.md` 必须同步更新

### 4.3 边界情况

- 衍生作品作为原作的"前传" / "后传" = 跨作品联动 = 需显式声明
- 衍生作品与原作同时推进同一伏笔（多视角呈现）= 显式声明后可允许

## 5. 违规判定流程

```
1. 加载 parent_canon.md
2. 检查所有具体陈述是否与原作矛盾
3. 检查所有"角色知道"是否在衍生时点合法
4. 检查所有规则使用是否符合 shared / spinoff-specific 规则
5. 检查所有伏笔操作范围是否正确
6. 任一违反 → 报告 + 严重度判定
```

## 6. 与原作协作文档建议

### 6.1 parent_canon.md 必备字段

```markdown
# 原作信息（衍生作品引用基础）

## 原作基本
- name: [原作名]
- author: [原作者]
- span: [故事发生时间范围]

## 关键事件表
- event-001: [事件描述] @ [原作章节 / 时点]
- event-002: ...

## 角色状态时点
- char-A: 状态变化轨迹
- char-B: ...

## 信息揭示时点
- info-001: 揭示内容 @ 时点
- info-002: ...

## 共享世界规则
- [与 world/rules.md 对应的关键规则]

## 跨作品联动声明（如有）
- linkage-001: 描述
```

### 6.2 维护

- 衍生作品每章审计前，先检查 `parent_canon.md` 是否更新
- 原作出新版时，需更新 `parent_canon.md` 并重审衍生近 5 章

## 7. 严重度统一

所有上述违规统一为 error 级别（无 warning）— 衍生作品的一致性是叙事契约的核心，违规 = 信任崩塌。
