# Review Evidence Format

所有 review/bug-hunt/clean skill 遵循此证据规范。每条缺陷/发现报告必须包含以下四要素，缺失任一即不合格。

## 四要素

1. **位置** — `文件路径` L起始行-结束行（如 `chapters/chapter-5.md` L23-27）
2. **原文引述** — 用 `>` 标记引述原文，≥20 字上下文
3. **违反规则** — 引用 SKILL.md 中的精确规则名（逐字匹配）
4. **严重度** — BLOCKING | CRITICAL | MINOR

## 严重度定义

| 级别 | 判定标准 | 示例 |
|------|---------|------|
| BLOCKING | 角色OOC、设定矛盾、连续性断裂、事实错误 | 角色第5章说"我不会打架"但第12章展示格斗技巧 |
| CRITICAL | AI味、风格漂移、节奏崩塌、对话声音不符 | 连续3个"不是…而是…"句式 |
| MINOR | 措辞不当、标点问题、单处冗余 | "了"字连续6句出现 |

## 输出模板

```markdown
## 缺陷 N

- **位置**: `chapters/chapter-5.md` L23-27
- **原文**: > 他缓缓地站起身来，似乎有什么东西在他的眼神中闪烁着，仿佛...
- **违反规则**: 铁律 #3 "AI标记词零出现"（缓缓地/似乎/仿佛 三词同段出现）
- **严重度**: CRITICAL
```

## bug-hunt 特殊要求

- 每条 planted defect 必须被检出并在报告中出现
- 检出位置的行号必须与注入位置一致（±2 行容差）
- 规则名必须与 SKILL.md 中的规则名精确匹配——近义词不算

## clean 特殊要求

- 逐文件确认：每个输出文件必须在报告中有对应确认条目
- 零问题声明：报告末尾必须有 "All files checked: 0 issues found"
- 禁止"改进建议"——clean 报告只陈述事实，改进建议 = 幻觉缺陷
